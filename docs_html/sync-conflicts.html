Conflict Resolution

ServerWins · LocalWins · LastWriteWins · Custom

A conflict occurs when a record was modified both locally and remotely since the last sync. syncitron supports four strategies, configured per table.

SyncStrategy Enum

StrategyBehaviourUse Case
serverWinsRemote record overwrites local. Default.Reference data, admin settings
localWinsLocal record is kept; remote update ignored.Drafts, user preferences
lastWriteWinsRecord with the latest updated_at wins.Collaborative content
customYour custom resolver function runs.Complex merge logic

Setting the Strategy

// Per table
engine.registerTable(TableConfig(
  name: 'settings',
  columns: ['id', 'key', 'value', 'updated_at', 'deleted_at'],
  strategy: SyncStrategy.serverWins,
));

Custom Conflict Resolver

When SyncStrategy.custom is selected, you must provide a customResolver. The resolver receives both records and returns a ConflictResolution.

engine.registerTable(TableConfig(
  name: 'documents',
  columns: ['id', 'title', 'body', 'version', 'updated_at', 'deleted_at'],
  strategy: SyncStrategy.custom,
  customResolver: (local, remote) async {
    // Keep the one with the higher version
    if ((local['version'] as int) >= (remote['version'] as int)) {
      return const UseLocal();
    }
    return UseRemote(remote);
  },
));

ConflictResolution Types

TypeMeaning
UseLocal()Keep the local dirty record as-is.
UseRemote(data)Overwrite local with the remote record data.
UseMerged(data)Save a manually merged map combining both versions.
🚫
Using SyncStrategy.custom without providing a customResolver will throw an EngineConfigurationException at registration time.

How Conflicts Are Detected

During the pull phase, for each incoming remote record, the engine checks if a local record with the same primary key exists and has is_synced = 0 (dirty). If so, it's a conflict. If the local record is already synced, the remote version is simply upserted.
